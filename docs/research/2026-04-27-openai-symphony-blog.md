---

## 背景

六个月前，团队在开发一款内部效率工具时做了一个颇具争议的决定：**整个代码仓库不允许出现任何人工编写的代码**，每一行代码都必须由 Codex 生成。

为此，团队从零重新设计了工程工作流，打造了一个对智能体友好的代码仓库，大力投入自动化测试与保护机制，并将 Codex 当作正式的团队成员来使用（详见之前关于 Harness Engineering 的博客文章）。

这个方案奏效了，但随即遇到了新瓶颈：**上下文切换**。

---

## 解决方案：Symphony

为解决这一问题，团队构建了 **Symphony** 系统。[Symphony](https://github.com/openai/symphony) 是一款智能体编排系统，它将 Linear 这样的项目管理看板转化为编码智能体的控制平面：每个未完成的任务都会分配一个智能体，智能体持续运行，由人类审查结果。

---

## 交互式编码智能体的瓶颈

随着 OpenAI 内部智能体工作规模的扩大，工程师们发现了新的负担：每位工程师同时管理三到五个 Codex 会话，一旦超出这个数量，频繁的上下文切换就会让效率大幅下降——遗忘哪个会话在做什么、在终端间来回切换、调试卡住的长任务。

**根本问题**：智能体速度快，但人类注意力是系统瓶颈。工程师实际上成了在微观管理一支能力超强的"初级工程师团队"，这根本无法规模化。

---

## 视角转变

团队意识到方向搞错了：过去围绕"编码会话"和"合并 PR"来组织系统，但这些只是手段，而非目标。软件工作流真正围绕的是可交付成果：**issue、任务、工单、里程碑**。

于是他们反思：如果停止直接监督智能体，而是让智能体自己从任务追踪器中拉取工作，会怎样？

这个想法催生了 Symphony——一份作为"监督者"的书面规范，用于编排智能体工作。

---

## 将 Issue 追踪器变为智能体编排器

Symphony 的核心理念很简单：**任何未关闭的任务都应由智能体接手并完成**。

在这一模式下，每一个未关闭的 Linear issue 都映射到一个专属的智能体工作空间。Symphony 持续监控任务看板，确保每一个活跃任务在完成之前始终有智能体在循环执行。如果某个智能体崩溃或卡住，Symphony 会自动重启它；如果出现新的工作，Symphony 会立即接手并开始组织执行。

工作流基于工单状态构建，将 Linear 作为一个状态机使用。

实践中，Symphony 将工作从会话和 PR 中解耦。有些 issue 会在多个代码仓库中产生多个 PR；另一些则纯粹是调研或分析，完全不涉及代码库。

> 一旦以这种方式对工作进行抽象，工单就可以代表更大粒度的工作单元。

团队会用 Symphony 编排复杂功能和基础设施迁移：比如让智能体分析代码库、Slack 或 Notion 并生成实施计划，然后由智能体生成任务树，将工作分解为多个阶段并定义依赖关系。

智能体只会处理未被阻塞的任务，因此对于有向无环图（DAG），执行会自然地并行展开。

智能体也可以自己创建新工作：在实施或评审过程中，它们往往能发现当前任务范围之外的改进点，并直接提交新 issue 供后续评估。

由于编排器运行在开发机上，永不休眠，工程师可以随时随地添加任务。比如，一位工程师曾在山间小屋用糟糕的 wifi，通过手机上的 Linear App 完成了三项重要变更。

---

## 效果：工作方式的增量探索

在 Symphony 带来的效果中，最明显的变化是**产出量**：在 OpenAI 某些团队中，头三周内成功合并的 PR 数量增加了 **500%**。

更深层的变化在于团队思考工作的方式：

- 工程师不再花时间监督 Codex 会话，每次代码变更的感知成本大幅下降
- 在 Symphony 中启动探索性任务变得轻而易举：试验想法、探索重构、验证假设，只保留有价值的结果
- **扩展了工作发起者的范围**：产品经理和设计师现在可以直接向 Symphony 提交功能请求，无需 checkout 代码仓库或管理 Codex 会话，只需描述功能，即可获得包含产品内功能演示视频的评审包
- 在大型 Monorepo 中，Symphony 还会监视 CI、在需要时 rebase、解决冲突、重试不稳定的检查，将变更推送合并的最后一公里全程自动化

---

## 进步带来的新问题

从交互式监督智能体转向在工单层面分配任务后，团队失去了即时干预和调整方向的能力，有时智能体会完全偏离目标。这些失败本身也很有价值——它们暴露了系统的漏洞，促使团队不断完善。

并非每个任务都适合 Symphony 的工作方式。有些问题仍然需要工程师直接与交互式 Codex 会话配合——尤其是模糊的问题，或需要深度判断和专业知识的工作。实际上，这些恰恰是工程师们最感兴趣、最愿意深入钻研的任务。

区别在于：**Symphony 可以承担绝大多数日常实施工作**，让工程师可以专注于一个真正困难的问题，而不是在大量小任务之间频繁切换。

团队还认识到，将智能体视为状态机中的僵化节点效果并不好。模型越来越聪明，所能解决的问题早已超出框定的范围。最终，团队转向给智能体分配**目标**，而非严格的状态转换——就像一个优秀的管理者会给下属分配目标一样。

## 用 Symphony 构建 Symphony

[Symphony 仓库](https://github.com/openai/symphony)的核心，本质上就是一个 `SPEC.md` 文件——对问题和预期解决方案的定义，而非一套复杂的监督系统。

参考实现使用 Elixir 编写（因为"代码几乎是免费的"，终于可以为其优势而选择语言，比如 Elixir 出色的并发性）。但核心理念可以用一个简单的 Markdown 文档来表达。

Codex 一次性完成了 Elixir 实现，之后团队在规范和实现上持续迭代。为了打磨规范，他们甚至让 Codex 用多种语言实现——TypeScript、Go、Rust、Java、Python——以此发现歧义并简化系统，每种语言都成功实现了。

整个开发工作流——处理 issue、checkout 代码仓库、更新进度状态、添加 PR、移至 **Review** 状态、附上演示视频等——现在都被记录在一个简单的 `WORKFLOW.md` 文件中，Symphony 确保智能体遵循这套流程。

> 对于每一个未关闭的任务，都确保有一个智能体在其独立工作空间中运行。

---

## Codex App Server 模式

团队使用了 Codex 的 [App Server 模式](https://developers.openai.com/codex/app-server/)——一种内置的无头运行模式，可通过完备的 JSON-RPC API 以编程方式与 Codex 交互（如启动线程、响应轮次）。

为避免将 Linear 访问令牌暴露给子智能体，团队使用了[动态工具调用](https://developers.openai.com/codex/app-server/#dynamic-tool-calls-experimental)来开放原始的 `linear_graphql` 函数，无需借助 MCP 或将令牌暴露给容器。

---

## 下一步

Symphony 是一个**刻意保持精简的编排层**，以开源形式发布，展示 Codex App Server 与 Linear 等工作流工具结合时的能量。

团队不打算将 Symphony 作为独立产品维护，而将其视为**参考实现**。希望开发者能将自己喜欢的编码智能体指向 Symphony 的[规范](https://github.com/openai/symphony/blob/main/SPEC.md)和[仓库](https://github.com/openai/symphony)，构建适合自己环境的版本。

随着编码智能体在推理和遵循指令方面越来越强，其他公司的瓶颈也将从编写代码转向管理智能体工作。令人兴奋的是，实验这些编码智能体系统的门槛已经出奇地低——**你完全可以用 Codex 来构建**。
